## 1. 产品定位更新（Web应用方向）

### 1.1 定位调整说明

| 维度 | 原PRD（移动优先） | 本文档（Web优先） |
|------|-------------------|-------------------|
| **核心平台** | iOS + Android App | **Web应用（浏览器访问）** |
| **Web端定位** | 轻量补充版 | **核心产品，全功能** |
| **用户入口** | 应用商店下载 | **浏览器URL直接访问，零安装** |
| **跨设备策略** | 双端原生App+iOS/Android一致 | **响应式Web，PC/平板/手机浏览器自适应** |
| **后续移动端** | MVP即上线 | **待定（PWA or 原生App，见决策点1）** |
| **核心交互范式** | 触屏操作、语音输入、推送通知 | **键鼠操作、拖拽上传、键盘快捷键、Web通知** |

### 1.2 保留不变的核心定位

> **"AI驱动的家庭协同财务管家"**  
> 核心价值主张：让家庭记账变成"不需要刻意记"的事——AI自动归集全渠道消费数据，夫妻共享一本账，智能洞察帮你管好钱。

三大产品目标不变：
1. **降低记账摩擦，突破留存天花板** — 目标不变（D7≥55%、D30≥35%）
2. **从"记录"升级到"建议"，建立AI财务洞察壁垒** — 目标不变
3. **占领家庭财务协同蓝海** — 目标不变

### 1.3 Web方向的核心理由

1. **零安装零更新**：用户打开浏览器即用，大幅降低获客和留存门槛（原PRD关注留存，Web天然降低进入摩擦）
2. **家庭协同天然适配**：大屏多窗口、键盘操作、多人可同时在不同设备访问同一账本
3. **账单导入更高效**：PC端用户本就在电脑上导出银行/支付宝账单，Web端直接拖拽上传，无需手机传文件
4. **开发成本更低**：一套代码全平台覆盖（响应式），无需维护iOS/Android双端
5. **财智用户迁移路径**：调研报告显示财智30-80万MAU是PC重度用户，面临"无处可迁"困境——Web应用是最佳承接产品

## 3. 关键流程的Web端交互方案

### 3.1 账单智能导入流程（Web端增强版）

```
用户点击导航栏"导入账单"或首页快捷入口
→ 进入导入页面，左侧为【拖拽上传区】（支持多文件批量拖入）
→ 右侧为【平台选择】：支付宝/微信/银行/信用卡（自动识别或手动选择）
→ 文件上传后显示进度条 → 解析完成自动跳转预览
→ 【表格预览页】：
   - 表格展示所有解析出的交易（日期/商户/金额/AI分类/状态）
   - 支持批量勾选 + 批量修改分类（右键菜单或顶部工具栏）
   - 支持列排序、筛选、搜索
   - 异常行高亮标记（金额异常/分类置信度低）
→ 用户确认导入 → 显示导入摘要（共X条，分类准确率Y%）
→ 数据合并至家庭账本/个人子账本
```

**Web端优势**：
- 拖拽多文件批量上传（移动端只能逐个选择）
- 表格视图批量操作（移动端逐条滑动操作效率低）
- 大屏预览所有记录，一目了然

**关键指标**：全流程<30秒/月（与原PRD一致），批量纠正效率提升50%+

### 3.2 语音/对话式记账流程（Web端适配版）

```
方式A — 键盘快捷键唤起（推荐）：
  用户在任意页面按 Cmd/Ctrl+K
  → 屏幕中央弹出【快速记账浮层】（类似Spotlight/命令面板）
  → 输入框获得焦点，用户直接打字："午饭28块" / "打车35元去公司"
  → 回车提交 → NLP解析 → 生成记录
  → 右下角Toast："已记：餐饮 -28元 [撤销]"
  → 继续输入或按Esc关闭

方式B — 语音输入（Chrome/Edge）：
  用户点击首页悬浮麦克风按钮
  → 浏览器请求麦克风权限（首次）
  → 用户说话："午饭花了28块"
  → Web Speech API实时转文字（<1秒）
  → NLP解析 → 生成记录 → Toast展示

方式C — 鼠标点击输入框：
  首页顶部常驻【记账输入框】，点击直接输入
```

**Web端差异**：
- 方式A（快捷键）是Web端独有优势，比移动端点击按钮更快
- 方式B依赖Web Speech API，Safari/Firefox支持有限，需降级提示"请使用Chrome或切换文字输入"
- 全流程<3秒（文字）/ <5秒（语音），与原PRD一致

### 3.3 AI财务洞察月报流程（Web端增强版）

```
每月1日系统自动生成月报 → 站内消息+邮件通知用户
→ 用户点击通知进入【月报页面】
→ 大屏报告布局：
   - 顶部：月度收支总览卡片（收入/支出/结余/环比）
   - 左侧：侧边目录导航（收支总览/消费结构/异常支出/建议/目标进度）
   - 主区域：当前章节内容（图表+自然语言解读）
   - 每条建议卡片带【立即处理】按钮 → 跳转对应功能页
   - 底部：【这条建议有帮助吗？】反馈收集
→ 支持【打印】【导出PDF】按钮
→ 支持【分享给家庭成员】（家庭成员登录后可查看）
```

**Web端优势**：
- 大屏报告阅读体验远超移动端
- 侧边目录导航快速跳转
- 打印/导出PDF便于家庭存档
- 家庭成员共享查看更方便

### 3.4 家庭账本创建与协同流程（Web端增强版）

```
用户点击"创建家庭账本" → 输入家庭名称
→ 系统生成【邀请链接】（可直接复制）+【邀请二维码】（可扫码加入）
→ 用户通过微信/邮件发送链接给家人
→ 家人点击链接 → 浏览器打开 → 注册/登录 → 自动加入家庭
→ 创建者进入【成员管理页】设置角色（管理员/普通成员/只读）
→ 各成员记账时选择记入"共同账本"或"个人子账本"（Tab切换）

协同视图：
  - 【家庭仪表盘】：聚合全家收支数据，大屏展示
  - 【成员视图】：按成员筛选查看各人记账情况
  - 【在线状态】：显示当前哪些家庭成员在线（绿点标识）
  - 【实时同步】：一方记账，其他成员页面实时刷新（WebSocket）
```

**Web端优势**：
- 大屏家庭仪表盘信息密度高
- 多Tab/多窗口可同时查看不同视图
- 邀请链接复制发送比移动端分享更方便
- 实时协同（WebSocket）在Web端是成熟技术

### 3.5 智能预算与温和预警流程（Web端适配版）

```
用户首次进入预算功能
→ AI分析历史3个月数据 → 推荐各分类预算金额
→ 【预算设置页】：表单+滑块组合，AI推荐值以灰色占位符显示
→ 用户确认或调整 → 关联心愿目标（如"日本旅行基金¥20000"）
→ 日常记账实时更新预算执行进度（首页仪表盘预算进度环）

预警触发（温和正向）：
  → 80%：Web通知 + 邮件"已用80%，离目标还差一点，加油"
  → 100%：Web通知 + 邮件"本月预算已用完，但旅行基金已存¥XXX"
  → 未超预算月末：邮件庆祝"本月省了¥XXX，旅行基金完成度XX%"

【站内消息中心】：所有预警历史可查看、可标记已读
→ 月末预算执行报告（含下月AI调整建议）
```

**Web端差异**：
- 移动推送 → Web通知（需用户授权）+邮件
- 站内消息中心集中管理所有通知
- 预算看板大屏展示更直观

### 3.6 新用户Onboarding流程（Web端适配版）

```
首次访问网站 → 欢迎页（1屏，<3秒）
→ 注册/登录（手机号+验证码 或 微信扫码登录 或 邮箱注册，≤2步）
→ 选择模式："个人记账" 或 "家庭记账"
→ 引导体验（步骤卡片+进度条，非强制可跳过）：
   1. "试试按Cmd+K快速记账" → 引导快捷键记账
   2. "导入你的支付宝账单" → 引导拖拽上传
   3. "设置你的第一个心愿目标" → 引导预算+目标
→ 进入首页（大屏仪表盘）
```

**Web端差异**：
- 增加微信扫码登录（Web端独有，移动端用微信SDK）
- 引导语音记账改为引导快捷键记账
- 首页大屏仪表盘给用户更强的"掌控感"第一印象

## 5. 技术方向建议（供架构师参考）

> **声明**：以下为产品经理视角的技术方向建议，具体架构设计由架构师负责。

### 5.1 前端技术栈建议

| 方面 | 建议 | 理由 |
|------|------|------|
| **框架** | React 18+ + TypeScript | 生态成熟、人才储备充足、复杂状态管理能力强 |
| **构建工具** | Vite | 开发体验好、HMR快、生产构建优化好 |
| **UI组件库** | shadcn/ui + Tailwind CSS 或 MUI | shadcn/ui灵活可定制、Tailwind快速开发；MUI组件丰富适合后台管理类页面 |
| **状态管理** | Zustand 或 Jotai（轻量）+ React Query（服务端状态） | 家庭协同场景需实时状态同步，React Query处理缓存和同步成熟 |
| **图表库** | Recharts 或 ECharts | ECharts功能强大适合大屏仪表盘；Recharts与React集成更自然 |
| **表格组件** | TanStack Table (React Table v8) | 支持排序/筛选/分组/虚拟滚动，适合大量交易数据展示 |
| **实时通信** | WebSocket（Socket.IO或原生WS） | 家庭协同实时同步必需 |

### 5.2 后端技术栈建议

| 方面 | 建议 | 理由 |
|------|------|------|
| **语言/框架** | Node.js (NestJS) 或 Python (FastAPI) | NestJS结构化好、TypeScript全栈统一；FastAPI适合AI密集场景 |
| **API风格** | RESTful + WebSocket（实时同步） | RESTful用于CRUD，WebSocket用于家庭协同实时推送 |
| **认证** | JWT + Refresh Token；支持微信扫码/手机号/邮箱 | Web端多登录方式，微信扫码是国内用户习惯 |
| **文件上传** | 分片上传 + 对象存储（OSS/S3） | 账单文件可能较大，分片上传提升体验 |

### 5.3 AI能力集成方案

| AI能力 | 集成方案 | 说明 |
|--------|----------|------|
| **账单解析** | 规则引擎 + NLP混合 | 先规则匹配格式，再NLP提取异常项；可配置解析器适配不同银行格式 |
| **语音识别** | Web Speech API（浏览器端）+ 云ASR降级（腾讯云/科大讯飞） | Chrome原生支持Web Speech API；不支持的浏览器降级到云端ASR |
| **OCR识别** | 云OCR API（腾讯云/百度云） | 票据识别用云API，不自研；按量付费成本可控 |
| **AI分类** | 规则+ML混合模型，云端推理 | Phase 0标注1000+条训练数据建立baseline；用户纠正反馈持续学习 |
| **AI洞察/建议** | 大语言模型API（如GPT-4/文心一言/通义千问）+ 规则模板 | 月报生成用LLM+模板组合，确保建议可执行且有数据支撑 |
| **AI对话顾问** | LLM + RAG（检索增强生成） | 基于用户真实财务数据回答，RAG确保回答有据可依 |

### 5.4 数据存储方案

| 存储类型 | 建议 | 用途 |
|----------|------|------|
| **主数据库** | PostgreSQL | 交易记录、用户数据、家庭账本、预算等核心数据；支持JSON字段灵活扩展 |
| **缓存** | Redis | 会话管理、实时协同状态、预算计算缓存、限流 |
| **文件存储** | 对象存储（阿里云OSS / AWS S3） | 上传的账单文件、导出的报告文件 |
| **全文搜索** | PostgreSQL全文检索 或 Elasticsearch（后续） | 交易记录搜索、商户名匹配；MVP用PG内置即可 |

### 5.5 部署与基础设施建议

| 方面 | 建议 |
|------|------|
| **部署方式** | 容器化部署（Docker + 云容器服务） |
| **CDN** | 静态资源CDN加速，提升全国访问速度 |
| **HTTPS** | 全站HTTPS（Let's Encrypt或云证书），Web Speech API和getUserMedia强制要求HTTPS |
| **监控** | 前端错误监控（Sentry）+ 后端APM + 用户行为埋点 |
| **数据合规** | 用户数据存储在中国境内服务器（如阿里云华东区），满足数据安全要求 |

## 6. 待用户确认的关键决策点

> 以下8个决策点需要用户拍板，将直接影响产品方向和开发范围。

### 决策点1：移动端策略 ⭐最高优先级

**问题**：Web应用为核心后，是否还需要移动端？如果需要，采用什么方案？

| 选项 | 优势 | 劣势 |
|------|------|------|
| A. 仅Web（响应式） | 开发成本最低，一套代码全平台 | 移动端浏览器体验受限（如语音记账、推送通知） |
| B. Web + PWA | 可安装到手机桌面、支持离线缓存、Web Push | iOS对PWA支持有限，体验仍不如原生 |
| C. Web优先，后续做原生App | 先验证Web产品，再决定是否投入原生 | 原生App开发成本高、周期长 |
| D. Web + 小程序 | 小程序覆盖微信生态，获客成本低 | 小程序功能受限，需额外维护 |

**产品经理建议**：选项B（Web + PWA），MVP阶段先做响应式Web，同步支持PWA安装和离线缓存。若用户反馈移动端体验不足，再考虑原生App。

---

### 决策点2：浏览器兼容范围

**问题**：需要兼容哪些浏览器？直接影响开发成本和功能可用性。

| 选项 | 兼容范围 | 影响 |
|------|----------|------|
| A. Chrome/Edge优先 | 仅保证Chrome/Edge最新版体验最佳 | 开发成本最低，Web Speech API等特性可用；但Firefox/Safari用户体验降级 |
| B. 主流浏览器全覆盖 | Chrome/Edge/Firefox/Safari最新版 | 开发成本中等，需处理兼容性降级；Safari对Web Speech API支持有限 |
| C. 含IE/旧版浏览器 | 向下兼容旧版 | 开发成本高，不推荐 |

**产品经理建议**：选项B，主流浏览器全覆盖但Chrome/Edge为优先适配目标。Safari降级方案：语音记账降级为文字输入。

---

### 决策点3：语音记账的优先级

**问题**：语音记账在Web端体验不如移动端（Web Speech API兼容性、无网络不可用），是否调整优先级？

| 选项 | 说明 |
|------|------|
| A. 维持P0，全力实现 | 投入资源解决兼容性问题，Chrome/Edge用Web Speech API，其他浏览器降级 |
| B. 降为P1，文字记账为主 | 以`Cmd/Ctrl+K`快捷键文字记账为核心卖点，语音作为增强功能延后 |
| C. 降为P2，后续再评估 | MVP完全不做语音，专注账单导入+文字记账+AI分类 |

**产品经理建议**：选项B，文字快捷记账（`Cmd/Ctrl+K`）是Web端更适合的极速记账方案，语音记账降为P1在Phase 2实现。

---

### 决策点4：通知机制方案

**问题**：移动端推送通知在Web端不可用，如何替代？

| 通知类型 | 建议方案 |
|----------|----------|
| 预算预警 | Web Notifications API（需授权）+ 站内消息中心 |
| 大额支出通知 | Web Notifications + 邮件 |
| 月报/周报推送 | 邮件为主 + 站内消息 |
| 家庭成员动态 | 站内消息中心 + 实时WebSocket推送（在线时） |

**需确认**：邮件通知是否需要集成邮件服务（如阿里云邮件推送）？用户是否接受邮件作为主要通知渠道？

**产品经理建议**：采用Web Notifications + 站内消息中心为主，邮件为辅。需集成邮件服务。

---

### 决策点5：实时协同的技术要求

**问题**：家庭协同场景对实时性要求多高？影响WebSocket vs 轮询的技术选型。

| 选项 | 说明 |
|------|------|
| A. 实时同步（WebSocket） | 一方记账，其他在线成员<3秒看到更新；技术成本高但体验好 |
| B. 准实时同步（30秒轮询） | 每30秒拉取一次更新；技术简单但体验有延迟 |
| C. 手动刷新 | 用户手动刷新页面查看最新数据；最简单但体验差 |

**产品经理建议**：选项A（WebSocket），家庭协同是核心差异化卖点，实时同步是刚需体验。原PRD要求同步延迟<3秒，仅WebSocket可满足。

---

### 决策点6：数据安全与合规要求

**问题**：用户财务数据存储和传输有哪些合规要求？

| 需确认项 | 影响 |
|----------|------|
| 数据是否必须存储在中国境内？ | 影响云服务商选择（阿里云/腾讯云 vs AWS） |
| 是否需要等保认证？ | 影响架构设计和运维成本 |
| 银行账单导入是否涉及金融数据合规审查？ | 影响P0-03是否可上线 |
| 是否需要提供数据本地化/私有部署选项？ | 影响产品定位（SaaS vs 可私有部署） |

**产品经理建议**：MVP阶段数据存储在中国境内云服务器（阿里云华东区），满足基本数据安全要求。等保认证和私有部署作为后续规划。

---

### 决策点7：商业化模式是否调整

**问题**：原PRD的"免费增值+订阅制（年付99-168元）"在Web平台是否需要调整？

| 考虑因素 | 说明 |
|----------|------|
| Web产品付费习惯 | 国内用户对Web工具付费意愿可能低于App（参考SaaS产品） |
| 免费版功能边界 | 原PRD未明确免费版包含哪些功能，需定义 |
| 付费点设计 | AI洞察报告、AI对话顾问、高级报表等可作为付费功能 |

**产品经理建议**：维持订阅制，但需重新定义免费版/付费版功能边界。建议：基础记账+账单导入+家庭共享免费；AI洞察+AI顾问+高级报表付费。

---

### 决策点8：MVP范围是否调整

**问题**：从移动优先转为Web优先后，MVP的15个P0需求是否需要调整范围？

| 建议调整 | 理由 |
|----------|------|
| P0-04语音记账 → 降为P1 | Web端体验受限，以快捷键文字记账替代 |
| W-01快捷键系统 → 提升为P0 | Web端核心交互，替代语音记账的极速记账方案 |
| W-02大屏仪表盘 → 提升为P0 | Web端核心差异化，首页第一印象 |
| W-03批量操作 → 提升为P0 | 账单导入和月底对账场景刚需 |

**产品经理建议**：MVP调整为14个P0（原15个减去语音记账，加上快捷键系统、大屏仪表盘、批量操作，净增2个）。总工期影响待架构师评估。
